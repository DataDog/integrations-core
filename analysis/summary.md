| Elasticsearch (`elastic`) | `url` | http-path-probe — The check only requires a URL (default `http://localhost:9200`) — Elasticsearch's REST API is on a well-known port. However, virtually all production deployments enable xpack security and require basic auth and TLS, so a discovery layer must (a) probe the port for the ES root document, (b) detect whether security is enabled, and (c) defer to the existing auto_conf template for the credentialed path. | HAProxy (`haproxy`) | — | http-path-probe — HAProxy supports two collection modes: a Prometheus exporter (HAProxy 2+ exposes `/metrics` on a configurable port, often 8404) which is unauthenticated and OpenMetrics-compatible, and the legacy stats page (`/admin?stats` or similar) which typically needs HTTP basic auth. A discovery layer must probe `/metrics` first and fall back to the stats page; classification is custom because of the multi-mode logic, but the OpenMetrics path is the easy case. | [spec](../haproxy/assets/configuration/spec.yaml), [upstream](https://www.haproxy.com/documentation/haproxy-configuration-tutorials/metrics-and-observability/prometheus/) |
| Kafka Consumer (`kafka_consumer`) | `kafka_connect_str` | other — The check connects to a Kafka broker bootstrap list (default port 9092 for plaintext, 9093 for SSL/SASL). Strictly per spec only `kafka_connect_str` is required, so unauthenticated localhost setups would discover via a Metadata API request. In practice, production Kafka uses SASL/PLAIN, SASL/SCRAM, or mTLS, all of which need credentials or certs that aren't on the wire. A discovery layer must probe the bootstrap port and either (a) succeed with PLAINTEXT or (b) note the protocol mismatch and require credentials. | [spec](../kafka_consumer/assets/configuration/spec.yaml), [upstream](https://kafka.apache.org/protocol) |
| MongoDB (`mongo`) | `hosts` | credentials-required — MongoDB's default port (27017) is universal and the wire protocol is detectable. Strictly per spec, only `hosts` is required, so an unauthenticated localhost dev instance could be discovered generically. In practice, every production MongoDB enables authentication and most also require a dedicated `datadog` user with `clusterMonitor` role, plus a `connection_scheme` decision (`mongodb` vs. `mongodb+srv`) for replica sets. Classification is custom because of the auth-mode branching, but in real deployments it falls through to credentials-required. | [spec](../mongo/assets/configuration/spec.yaml), [upstream](https://www.mongodb.com/docs/manual/tutorial/configure-x509-client-authentication/) |
| NGINX (`nginx`) | `nginx_status_url` | http-path-probe — NGINX has at least three plausible status paths depending on the build: stub_status (open-source nginx), the NGINX Plus `/api` (R13+), and the vhost_traffic_status (`vts`) module's JSON endpoint. The status path is operator-configurable, so a discovery layer must probe candidate paths and inspect the response shape to pick the right mode. The port itself is whatever the operator configured. | [spec](../nginx/assets/configuration/spec.yaml), [upstream](https://nginx.org/en/docs/http/ngx_http_stub_status_module.html), [upstream](https://docs.nginx.com/nginx/admin-guide/monitoring/live-activity-monitoring/) |

A discovery layer must probe the Prometheus port first and fall back to the Management API. Classification is custom because of the multi-mode logic; the Prometheus path alone would be generic. | [spec](../rabbitmq/assets/configuration/spec.yaml), [auto_conf](../rabbitmq/datadog_checks/rabbitmq/data/auto_conf.yaml), [upstream](https://www.rabbitmq.com/docs/prometheus) |

### Auto-config impossible

| Integration | Required fields | Method / detail | References |
|---|---|---|---|
| MySQL (`mysql`) | `host`, `username`, `password` | credentials-required — MySQL needs a dedicated `datadog` user with explicit grants and a password to read performance schema and replication state. The credentials cannot be discovered from the wire — they must be provisioned in the database first. | [spec](../mysql/assets/configuration/spec.yaml) |
| Postgres (`postgres`) | `host`, `port`, `username`, `password` | credentials-required — Postgres requires a dedicated `datadog` user with explicit grants and a password to read the statistics views. The credentials cannot be discovered from the wire; they must be provisioned out-of-band. Once provided, host/port have universal defaults but the auth blocker dominates classification. | [spec](../postgres/assets/configuration/spec.yaml) |
| SNMP (`snmp`) | `ip_address` | credentials-required — SNMP needs at minimum the device IP and a credential: a community string (v1/v2c) or a v3 user with auth/priv keys. Community strings could occasionally be guessed (`public`/`private`) but production devices have unique strings, and v3 is the modern default. The integration has its own network-discovery feature (`network_address` + profile auto-detection) but it still needs a credential to authenticate. Classification is impossible for credential discovery; SNMP-internal device discovery is orthogonal. | [spec](../snmp/assets/configuration/spec.yaml), [upstream](https://datatracker.ietf.org/doc/html/rfc3411) |
| SQL Server (`sqlserver`) | `host` | credentials-required — SQL Server requires either SQL authentication (username + password for a dedicated `datadog` user with `VIEW SERVER STATE` and other grants) or Windows authentication (the Agent's process identity must have those grants on the SQL Server instance). Either form is provisioned out-of-band; the credentials cannot be derived from the wire. Default port 1433 is universal but irrelevant given the auth blocker. | [spec](../sqlserver/assets/configuration/spec.yaml), [upstream](https://learn.microsoft.com/en-us/sql/relational-databases/system-dynamic-management-views/system-dynamic-management-views) |
