## 2. metadata.csv — column format and conventions

File: `/home/bits/dd/integrations-core/ibm_db2/metadata.csv`

### 2.1 Header / column order (exact)

```
metric_name,metric_type,interval,unit_name,per_unit_name,description,orientation,integration,short_name,curated_metric
```

The validator enforces this exact ordered set. Authoritative source:
`/home/bits/dd/integrations-core/ddev/src/ddev/cli/validate/metadata_utils.py`

- `REQUIRED_HEADERS = {'metric_name', 'metric_type', 'orientation', 'integration'}`
  (`metadata_utils.py:6`)
- `OPTIONAL_HEADERS = {'description', 'interval', 'unit_name', 'per_unit_name', 'short_name', 'curated_metric'}`
  (`metadata_utils.py:8-15`)
- `EXPERIMENTAL_HEADER = {"sample_tags"}` (`metadata_utils.py:17`) — exists but `ibm_db2`'s
  metadata.csv does NOT include it; do not add it unless intentionally adopting sample_tags.
- `ORDERED_HEADERS` (`metadata_utils.py:21-33`) defines the canonical column order shown above.

### 2.2 Allowed values

- **metric_type** (`metadata_utils.py:35`): `VALID_METRIC_TYPE = {'count', 'gauge', 'rate'}`.
  Only these three are valid in metadata.csv. IMPORTANT: the *check code* submits via
  `self.monotonic_count(...)` (a submission method on `AgentCheck`) but the metadata.csv
  `metric_type` for those metrics is written as **`count`** (see existing rows like
  `ibm_db2.connection.total,count,...`). So: code uses `gauge` / `monotonic_count` /
  `count` / `rate` methods; metadata.csv records `gauge` / `count` / `rate`. A
  `monotonic_count` submission is catalogued as `count`.
- **orientation** (`metadata_utils.py:37`): `VALID_ORIENTATION = {'0', '1', '-1'}`.
  `0` = neutral, `1` = higher is better, `-1` = lower is better. Existing Db2 examples:
  hit_percent → `1`, utilized/used → `-1`, reads → `0`, backup.latest → `-1`.
- **unit_name** (`metadata_utils.py:55-411`, `VALID_UNIT_NAMES`): large fixed allow-list. Units
  already used in this integration: `connection`, `second`, `percent`, `get`, `lock`, `page`,
  `millisecond`, `block`, `read`, `write`, `row`, `byte`. Other generally useful values present
  in the allow-list for DB metrics: `query`, `transaction`, `commit`, `wait`, `table`, `index`,
  `session`, `request`, `operation`, `hit`, `miss`, `eviction`, `nanosecond`, `microsecond`,
  `kibibyte`/`mebibyte`/`gibibyte` (binary), `kilobyte`/`megabyte`/`gigabyte` (decimal),
  `event`, `error`, `time`, `key`, `record`, `object`, `cursor`, `sort`, (note: `sort` is NOT in
  the list — verify against `VALID_UNIT_NAMES` before using any new unit).
- **per_unit_name**: same allow-list as `unit_name`; used to express rates like `byte` per
  `second`. Empty in all current Db2 rows.
- **interval**: empty in all current rows; only set for metrics with a fixed reporting interval.
- **integration**: must be `ibm_db2` for every row.
- **short_name**, **curated_metric**: empty in all current rows. `curated_metric` if set must be
  one of `VALID_CURATED_METRIC_TYPES = {'cpu', 'memory'}` (`metadata_utils.py:39`).

### 2.3 Other validation rules (same file)

- `MAX_DESCRIPTION_LENGTH = 400` (`metadata_utils.py:416`) — keep descriptions ≤ 400 chars.
- Descriptions containing commas must be double-quoted (CSV); existing rows do this, e.g. row
  for `ibm_db2.bufferpool.column.reads.logical`.
- Metric name validation regexes: `METRIC_REPLACEMENT` and `METRIC_DOTUNDERSCORE_CLEANUP`
  (`metadata_utils.py:418-419`). Names should be lowercase, dot-separated, `[a-z0-9_.]`.
- `ALLOWED_PREFIXES` (`metadata_utils.py:413`) are reserved system prefixes; Db2 metrics use the
  `ibm_db2.` prefix which is fine.

### 2.4 Existing metric namespace (for fidelity gap analysis)

All 50 current metrics are prefixed `ibm_db2.` and grouped:
`ibm_db2.application.*`, `ibm_db2.backup.latest`, `ibm_db2.bufferpool.*` (column/data/index/xda
plus group.* pureScale variants, each with reads.logical/physical/total + hit_percent),
`ibm_db2.connection.*` (active/max/total), `ibm_db2.lock.*` (active/dead/pages/timeouts/wait/
waiting), `ibm_db2.log.*` (available/reads/used/utilized/writes), `ibm_db2.row.*`
(modified.total/reads.total/returned.total), `ibm_db2.tablespace.*`
(size/usable/used/utilized). Full list with types/units is in metadata.csv lines 2-50; the test
manifest of expected metrics is `/home/bits/dd/integrations-core/ibm_db2/tests/metrics.py`
(lists `STANDARD`, `BUFFERPOOL`, `TABLESPACE`).

### 2.5 Prefix declared in manifest

`manifest.json` declares the metric prefix and a "check" canary metric:
```json
"metrics": { "prefix": "ibm_db2.", "check": "ibm_db2.connection.active", "metadata_path": "metadata.csv" }
```
(`/home/bits/dd/integrations-core/ibm_db2/manifest.json:46-50`). If you add metrics, the prefix
stays `ibm_db2.`; the `check` canary should remain a metric that is always emitted.

### 3.2 spec.yaml structure (current Db2)

`spec.yaml` top-level keys (`spec.yaml:1-3`): `name: IBM Db2`, `fleet_configurable: true`,
`files:`. There is a single file `ibm_db2.yaml` with `init_config` and `instances` sections built
from shared templates plus inline option overrides.

Templates referenced (resolved from
`/home/bits/dd/integrations-core/datadog_checks_dev/datadog_checks/dev/tooling/templates/configuration/`):
- `init_config` → `init_config.yaml`
- `init_config/db` → `init_config/db.yaml` (adds `global_custom_queries`)
- `init_config/default` → `init_config/default.yaml`
- `instances` → `instances.yaml`
- `instances/db` → `instances/db.yaml` (adds `only_custom_queries`, `use_global_custom_queries`,
  `custom_queries`)
- `instances/default` → `instances/default.yaml`
- `logs`

Inline instance options currently defined in spec.yaml (`spec.yaml:17-126`): `db` (required),
`username` (required), `password` (required, `secret: true`), `host`, `port` (default example
50000), `security` (enum `none`/`ssl`), `tls_cert`, `connection_timeout`. The `custom_queries`
template is overridden to document the 3-field Db2 custom-query format
(`spec.yaml:76-126`).

Each option supports keys observed in the spec / templates: `name`, `description`,
`fleet_configurable`, `required`, `secret`, `hidden`, `display_priority`, `value:` (with `type`,
`example`, `enum`, `display_default`), and `options:` (for nested/object options). See
`instances/db.yaml` for nested `value.items.properties` usage.

### 3.3 Generated pydantic model anatomy

`config_models/instance.py` (`InstanceConfig`, `:44-87`):
- `model_config = ConfigDict(validate_default=True, arbitrary_types_allowed=True, frozen=True)`.
- Fields are typed from spec value `type`s (e.g. `db: str`, `port: Optional[int]`,
  `security: Optional[Literal['none', 'ssl']]`). Note: required spec options (`db`, `username`,
  `password`) become non-Optional fields.
- `_initial_validation` (`mode='before'`) calls `validators.initialize_instance` if present.
- `_validate` (per-field, `mode='before'`) dispatches to `validators.instance_<field>` if the
  field was configured, else to `defaults.instance_<field>`.
- `_final_validation` (`mode='after'`) calls `validators.check_instance` if present.

`config_models/shared.py` (`SharedConfig`, `:23-47`): currently holds `global_custom_queries`
and `service`. Same validator dispatch pattern with `shared_`/`initialize_shared`/`check_shared`.

`config_models/defaults.py` (`:11-44`): one function per defaulted field, e.g.
`instance_connection_timeout()→10`, `instance_port()→50000`, `instance_security()→'none'`,
`instance_min_collection_interval()→15`, `instance_only_custom_queries()→False`,
`instance_use_global_custom_queries()→'true'`, `instance_disable_generic_tags()→False`,
`instance_empty_default_hostname()→False`, `instance_enable_legacy_tags_normalization()→True`.

### 3.4 How to wire new config in the check

The check class does NOT currently inherit `ConfigMixin`; it reads raw `self.instance.get(...)`
(see `IbmDb2Check.__init__`, `ibm_db2.py:35-66`). New DBM config can either continue that pattern
or adopt `ConfigMixin` (`config_models/__init__.py:14-24`) which exposes `self.config`
(`InstanceConfig`) and `self.shared_config`. For DBM-style nested config (e.g. `dbm`,
`query_metrics.enabled`, `query_activity.collection_interval`), follow the sqlserver/mysql
pattern of nested option objects in spec.yaml that generate nested pydantic models.

  `{added, changed, deprecated, removed, fixed, security}`. (Observed types in repo: `.added`,
  `.changed`, `.fixed`, `.removed`.) The fragment file content is a single short sentence (no
  trailing period required; markdown allowed). Examples:
  - `ibm_i/changelog.d/23986.changed` → `Change `ibm_i.can_connect` service check to report OK ...`

## 8. ddev — exact commands (validate / generate models / test / e2e)

`ddev` is a Click CLI. The global `-x`/`--here` flag targets the current integration directory
(`ddev/src/ddev/cli/__init__.py:37`). Below, run either with `-x` from inside `ibm_db2/`, or pass
`ibm_db2` from the repo root.

### 8.1 Validate (subcommands registered in `ddev/src/ddev/cli/validate/__init__.py:41-66`)

Registered validate subcommands: `agent-reqs`, `all`, `ci`, `codeowners`, `config`,
`dashboards`, `dep`, `eula`, `http`, `imports`, `integration-style`, `jmx-metrics`, `labeler`,
`legacy-signature`, `license-headers`, `licenses`, `metadata`, `models`, `openmetrics`,
`package`, `qa-label`, `readmes`, `saved-views`, `service-checks`, `typos`, `version`.

Key commands for metric + DBM work:
```bash
# Regenerate conf.yaml.example + sync from spec.yaml (run BEFORE models):
ddev validate config ibm_db2 --sync          # or: ddev -x validate config -s   (from ibm_db2/)

# Regenerate config_models/*.py from spec.yaml (run AFTER config):
ddev validate models ibm_db2 --sync          # or: ddev -x validate models -s   (from ibm_db2/)

# Validate metric catalog (metadata.csv columns/types/units/orientation):
ddev validate metadata ibm_db2

# Validate service_checks.json:
ddev validate service-checks ibm_db2

# Validate README anchors/links:
ddev validate readmes ibm_db2

# Validate dependencies / imports / license headers as needed:
ddev validate dep ibm_db2
ddev validate license-headers ibm_db2

# Run the full CI-equivalent validation suite for the integration:
ddev validate all ibm_db2
ddev validate ci ibm_db2
```
The `--sync` flag is what actually rewrites the generated files; without it the command only
checks for drift (and fails CI if drift exists). The header comment in generated files shows the
`-x ... -s` form where `-s` is the short alias for `--sync`
(`config_models/instance.py:6-8`).

Canonical two-step regeneration after editing spec.yaml:
```bash
ddev -x validate config -s      # writes data/conf.yaml.example
ddev -x validate models -s      # writes config_models/{instance,shared,defaults}.py
```

### 8.2 Test (unit / integration) — `ddev test`

`ddev test` is registered at `ddev/src/ddev/cli/test/__init__.py`. It drives the hatch test
environments defined in `hatch.toml`.
```bash
# Lint + unit + integration (default):
ddev test ibm_db2

# Only unit tests (pytest marker 'unit'):
ddev test ibm_db2 --marker unit          # alias: -m unit

# Only integration tests (spins up dd_environment docker via conftest):
ddev test ibm_db2 --marker integration

# Coverage / specific test selection:
ddev test ibm_db2 --cov
ddev test ibm_db2 -- -k test_standard    # pass-through args after `--`
```
Pytest markers used by Db2 tests: `pytest.mark.unit` (`tests/test_unit.py:11`),
`pytest.mark.integration` (`tests/test_integration.py:13`), and `pytest.mark.e2e`
(`tests/test_integration_e2e.py:21`). Integration tests require the `dd_environment` fixture
(`tests/conftest.py:65-70`) which uses `docker_run(COMPOSE_FILE, ...)`.

### 8.3 E2E — `ddev env`

The `ddev env` command group (`ddev/src/ddev/cli/env/`) manages a live Agent + datasource for
end-to-end runs. Subcommands present: `start`, `stop`, `check`, `agent`, `shell`, `reload`,
`show`, `config`, `test`.
```bash
# Start an e2e environment (uses tests/docker/docker-compose.yaml via dd_environment):
ddev env start ibm_db2 py3.13-12.1        # <env> is "<python>-<version>" from hatch matrix

# Run the check once against the live env:
ddev env check ibm_db2 py3.13-12.1

# Run the e2e-marked pytest tests against the running env:
ddev env test ibm_db2 py3.13-12.1         # runs @pytest.mark.e2e tests (dd_agent_check fixture)

# Tear down:
ddev env stop ibm_db2 py3.13-12.1
```
The e2e test entrypoint is `tests/test_integration_e2e.py::test_e2e` which uses the
`dd_agent_check` fixture (`tests/test_integration_e2e.py:20-24`). The `dd_environment` fixture
yields `(CONFIG, E2E_METADATA)`; `E2E_METADATA` (`tests/common.py:27-37`) installs `ibm_db`
inside the Agent container via `start_commands`.

## 11. DBM (Database Monitoring) conventions — reference pattern (net-new for Db2)

Db2 has no DBM module today. The canonical implementations to mirror are sqlserver, mysql,
postgres. Key building blocks (all in `datadog_checks_base`):

### 11.1 Payload submission methods (on `AgentCheck`)

`/home/bits/dd/integrations-core/datadog_checks_base/datadog_checks/base/checks/base.py:772-798`:
- `database_monitoring_query_sample(raw_event)` → event-platform track `"dbm-samples"`
- `database_monitoring_query_metrics(raw_event)` → track `"dbm-metrics"`
- `database_monitoring_query_activity(raw_event)` → track `"dbm-activity"`
- `database_monitoring_metadata(raw_event)` → track `"dbm-metadata"`
- generic `event_platform_event(raw_event, event_track_type)` (`:800`).
`raw_event` is a JSON string; producers serialize with
`json.dumps(payload, default=default_json_event_encoding)` (see mysql usage
`/home/bits/dd/integrations-core/mysql/datadog_checks/mysql/statements.py:163,176`).

### 11.2 Async job framework

`DBMAsyncJob` in
`/home/bits/dd/integrations-core/datadog_checks_base/datadog_checks/base/utils/db/utils.py:289`.
Constructor params (`:299-320`): `check`, `config_host`, `min_collection_interval`, `dbms`,
`rate_limit`, `max_sleep_chunk_s`, `run_sync`, `enabled`, `expected_db_exceptions`,
`shutdown_callback`, `job_name`, `enable_missed_collection_event`, `features`. A subclass
implements `run_job()` and the check calls `run_job_loop(tags)` each check run (`:354`). Host
resolution via `resolve_db_host()` (`utils.py:181`). Each Db2 DBM collector (query metrics,
activity/samples, settings, schemas) would be its own `DBMAsyncJob` subclass with a distinct
`job_name`/`dbms='db2'`.

### 11.3 Supporting base utilities

In `datadog_checks_base/datadog_checks/base/utils/db/`:
- `utils.py`: `ConstantRateLimiter` (`:106`), `RateLimitingTTLCache` (`:148`),
  `resolve_db_host` (`:181`), `default_json_event_encoding` (`:237`),
  `obfuscate_sql_with_metadata` (`:249`), `DBMAsyncJob` (`:289`).
- `statement_metrics.py` — diff/rate computation across collections for query metrics.
- `sql.py` / `sql_commenter.py` — SQL obfuscation + comment parsing (APM/DBM correlation).
- `schemas.py` — schema collection helpers.

### 11.4 Reference module layout (sqlserver / mysql)

- sqlserver (`/home/bits/dd/integrations-core/sqlserver/datadog_checks/sqlserver/`):
  `statements.py`, `activity.py`, `metadata.py`, `schemas.py`, `deadlocks.py`,
  `agent_history.py`, `xe_collection/`.
- mysql (`/home/bits/dd/integrations-core/mysql/datadog_checks/mysql/`):
  `statements.py`, `statement_samples.py`, `activity.py`, `metadata.py`.
For Db2, the analogous collectors would query the Db2 monitoring views/functions
(`MON_GET_PKG_CACHE_STMT` for statement metrics, `MON_GET_ACTIVITY`/`WLM_GET_*` for active
sessions/samples, `SYSCAT.*` for schema/settings metadata). Those exact Db2 sources are a
fidelity-plan decision, not yet present in this repo.

### 11.5 spec.yaml DBM options pattern

sqlserver spec (`/home/bits/dd/integrations-core/sqlserver/assets/configuration/spec.yaml`) is
the template for DBM config keys:
- `dbm` boolean (`spec.yaml:567-573`): "Set to `true` to enable Database Monitoring."
- Nested option objects each with `enabled` + `collection_interval`: `collect_settings`
  (`:575`), `query_metrics` (`:593`), `query_activity` (`:707`), `collect_schemas` (`:1093`).
- Cloud metadata blocks `aws`/`gcp`/`azure` (`:741`/`:763`/`:794`), `obfuscator_options`
  (`:868`), `propagate_agent_tags` (`:22`, `:1161`). Many carry the note "Requires `dbm: true`"
  or "only applies when `dbm` is enabled".
These become nested pydantic models after `ddev validate models --sync`.

### 11.6 dataflows.yaml for DBM

File: `/home/bits/dd/integrations-core/ibm_db2/assets/dataflows.yaml` currently declares only
`metrics`, `logs`, `events` (ids `ibm-db2-metrics`/`-logs`/`-events`). Each entry has `id`,
`always_on`, `data_type`, `direction: inbound`. Note: postgres/sqlserver/mysql dataflows.yaml
likewise only declare `metrics`/`logs`/`events` (postgres example confirmed:
`/home/bits/dd/integrations-core/postgres/assets/dataflows.yaml`) — DBM is NOT a separate
`data_type` in dataflows.yaml. DBM payloads go through the event-platform tracks in §11.1, not a
new dataflows entry.

### 11.7 metadata.csv for DBM

DBM query-level data (samples/activity/plans) is NOT catalogued in metadata.csv (it is
event-platform data, not Agent metrics). Only summary/health gauges/counts that the check emits
via `self.gauge`/`self.count` belong in metadata.csv. So a DBM addition typically adds few or no
metadata.csv rows; the bulk is event-platform payloads.

## 13. Concrete gaps / decisions surfaced for the implementation plan

1. **Db2 12.1 image/matrix**: `hatch.toml` matrix is `version=["11.1"]` and docker uses
   `taskana/db2:${DB2_VERSION}`. Targeting 12.1 requires (a) adding `"12.1"` to the matrix and
   (b) a docker image that publishes a 12.1 tag (taskana/db2 may not — likely switch to
   `icr.io/db2_community/db2` / `ibmcom/db2`). Live internal container is 12.1.4.
2. **DBM is net-new**: no DBM module, no `dbm`/`query_metrics`/`query_activity` spec options, no
   DBM service check, and `pyproject.toml` base pin (`>=37.33.0`) may need a bump for the DBM
   utilities. Mirror sqlserver/mysql module layout + spec blocks + `DBMAsyncJob`.
3. **Version parsing**: `parse_version` assumes `MM.mm.uuuu`; confirm 12.1.4's `SQL_DBMS_VER`
   format still parses (`ibm_db2.py:109-125`).
4. **metadata.csv vs code type**: remember `monotonic_count` submissions are catalogued as
   `count` in metadata.csv (only `count`/`gauge`/`rate` are valid there).
